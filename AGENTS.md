# AGENTS.md

## Назначение
Этот файл задаёт правила для AI-агентов (Codex/ChatGPT) при изменениях в репозитории `RAG-Test-OrVD`.

## Контекст проекта
- Тип системы: **offline-first RAG-платформа поддержки** (FastAPI + Qdrant + PostgreSQL + локальные модели).
- Ключевые API: `/ask`, `/v1/chat/completions`, `/ingest/a/run`, `/ingest/b/run`, `/sources/*`.
- Основная цель изменений: повышать надёжность retrieval/generation без ломки API-контрактов.

## Особенности системы (обязательно учитывать при изменениях)

### 1) Архитектурные особенности
- Система работает в локальном контуре: внешние API и облачные сервисы не должны быть обязательной зависимостью рантайма.
- Поддерживаются два независимых корпуса знаний:
  - `csv_ans_docs` (корпус A);
  - `internal_regulations` (корпус B).
- Индексация и поиск распределены между:
  - **Qdrant** — векторы/поиск по чанкам;
  - **PostgreSQL** — метаданные документов, диагностическая информация.
- Ответы формируются через orchestration-слой (`rag/*`), где последовательно выполняются retrieval → rerank (если включён) → prompt assembly → generation.

### 2) Модельный стек и мультимодальность
- Зафиксированный стек моделей менять только по отдельному согласованию (LLM, embeddings, reranker, OCR, VLM).
- Мультимодальный контур поддерживает изображения в runtime (`/ask`, `/v1/chat/completions`) и в ingestion (извлечение из DOCX/PDF).
- Vision-режимы (`ocr|vlm`) переключаются env-параметрами отдельно для runtime и ingest; при изменениях сохранять обратную совместимость дефолтного поведения.
- Поля `images` и `visual_evidence` считаются частью пользовательского контракта и не должны пропадать/менять тип без документированного миграционного решения.

### 3) Контракт API и совместимость
- `/ask` и `/v1/chat/completions` считаются критичными контрактами интеграции.
- Любые изменения схем запроса/ответа должны сопровождаться:
  1. обновлением `API_CONTRACTS.md`;
  2. обновлением integration/unit тестов;
  3. явным описанием рисков совместимости в PR.
- Для `sources` обязательно сохранять возможность построения/возврата `download_url` на исходные документы.

### 4) Ingestion и качество данных
- Ingestion A/B должен оставаться воспроизводимым и детерминированным в рамках текущих настроек chunking/dedup.
- Изменения chunking/retrieval/reranker требуют unit-тестов (`app/tests/unit`) и пояснения, почему изменение улучшает качество.
- Изменения ingestion обязаны учитывать оба корпуса (`csv_ans_docs`, `internal_regulations`) и их различия по chunking-стратегии.
- Если меняется обработка изображений/документов, фиксировать влияние на `diagnostics` ingest-ответа.

### 5) Операционные требования
- Не логировать секреты, токены, персональные данные и чувствимые фрагменты пользовательских вложений.
- Наблюдаемость (метрики/логи) не должна деградировать: при добавлении новых этапов пайплайна добавлять осмысленные диагностические события.
- Новые env-параметры:
  - добавлять в `app/src/core/settings.py`;
  - документировать в репозитории (`README.md` и/или профильный `docs/*`).

## Обязательный порядок работы
1. Прочитать `README.md`, `CODEBASE_MAP.md`, `ARCHITECTURE.md`, `API_CONTRACTS.md`.
2. При изменениях API или RAG-логики обновить документацию в этом репозитории.
3. Перед финализацией изменений выполнить:
   - `cd app && pytest -q`
   - при изменениях пайплайнов/мультимодальности: `python3 scripts/run_vision_regression.py --api-url http://localhost:8000` (если локальный стек поднят).

## Правила изменений
- Не добавлять облачные зависимости без явного запроса.
- Не менять зафиксированный модельный стек без отдельного согласования.
- Не ломать обратную совместимость ответов `/ask` и `/v1/chat/completions` (`sources`, `images`, `visual_evidence`).
- Не логировать секреты/PII.
- Новые env-параметры добавлять в `app/src/core/settings.py` и документировать.

## RAG-правила
- Изменения в chunking/retrieval/reranker сопровождаются unit-тестами (`app/tests/unit`).
- Изменения ingestion для корпусов A/B требуют проверки на обоих наборах (`csv_ans_docs`, `internal_regulations`).
- Prompt-изменения фиксировать в `PROMPTS.md`.

## Коммиты и PR
- Формат коммитов: `type(scope): summary` (например, `docs(project): add v1 context docs`).
- В PR описывать:
  - что изменено;
  - почему;
  - как проверено;
  - риски/ограничения.
